# 黑夜君临悬浮助手项目章程

## 核心原则

### I. 安全至上

**本程序必须保证游戏的安全性，这是不可妥协的底线。**

- 程序必须仅通过截屏方式获取游戏信息
- 严格禁止读取或修改游戏内存
- 严格禁止修改游戏数据文件
- 严格禁止注入游戏进程
- 所有功能实现必须经过安全性审查，确保不违反游戏使用条款

**理由**: 保护用户账号安全，维护游戏公平性，避免被游戏反作弊系统误判。

### II. 用户体验优先

**所有功能必须以提升用户游戏体验为目标，不能干扰正常游戏。**

- 悬浮窗必须可自定义位置、大小和透明度
- 所有快捷键必须可配置，避免与游戏快捷键冲突
- UI必须简洁直观，设置项必须有清晰的说明文档
- 性能开销必须最小化，不能影响游戏帧率
- 必须提供完整的中文界面和文档

**理由**: 作为辅助工具，不应成为游戏的负担，而应是无缝的辅助体验。

### III. 性能为本

**程序必须高效运行，资源占用必须控制在合理范围内。**

- CPU使用必须最小化，空闲时接近0%，活跃处理时应保持在较低水平
- 内存占用必须低于200MB
- 截图识别延迟必须低于100ms
- UI渲染必须保持60fps
- 必须使用异步处理避免阻塞主线程

**理由**: 游戏辅助工具不应占用大量系统资源，影响游戏性能。

### IV. 可维护性

**代码必须清晰、模块化，便于后续维护和功能扩展。**

- 代码必须遵循PEP 8规范
- 每个功能模块必须职责单一，高内聚低耦合
- 关键算法必须有注释说明
- 配置必须与代码分离，便于调整
- 图片资源必须统一管理，路径可配置

**理由**: 游戏会持续更新，辅助工具需要快速适配新版本。

### V. 测试驱动开发（推荐）

**重要功能应该有测试保障，但不强制TDD流程。**

- 新增的核心识别算法应该编写单元测试
- UI交互逻辑应该有集成测试
- 重要功能修改前应先添加回归测试
- 鼓励但不强制先写测试再实现的开发流程

**理由**: 确保代码质量，但考虑到个人项目的灵活性，不做强制要求。

## 技术约束

### 开发环境

- **语言版本**: Python 3.13
- **UI框架**: PyQt6
- **目标平台**: Windows 7/8/10/11
- **打包工具**: PyInstaller
- **图像处理**: OpenCV / PIL
- **OCR引擎**: PaddleOCR / Tesseract（如需要）

### 依赖管理

- 必须在requirements.txt中明确列出所有依赖及版本
- 打包时必须包含所有必需的运行时依赖
- 第三方库选择必须考虑许可证兼容性

### 版本控制

- 版本号格式: MAJOR.MINOR.PATCH
- MAJOR: 重大功能变更或不兼容的API修改
- MINOR: 新增功能，向后兼容
- PATCH: 问题修复和性能优化

## 开发流程

### 功能开发流程

1. **需求分析**: 明确功能目标和用户场景
2. **技术方案**: 评估实现方式，确保符合安全性原则
3. **原型开发**: 先实现核心功能，验证可行性
4. **性能优化**: 确保满足性能指标
5. **测试验证**: 在实际游戏环境中测试
6. **文档更新**: 更新README和设置界面的帮助文档

### 代码审查要点

- 是否违反安全性原则
- 是否影响程序性能
- 是否有明显的代码异味
- 是否有足够的错误处理
- 是否更新了相关文档

### 发布流程

1. 更新版本号
2. 更新CHANGELOG
3. 执行build.bat构建可执行文件
4. 在干净的Windows环境测试
5. 创建GitHub Release
6. 发布更新说明

## 治理规范

### 原则修订

- 章程修改必须有明确的理由和改进目标
- 重大修改（MAJOR版本）需要充分讨论
- 修改必须更新版本号和修订日期
- 修改后必须检查相关模板的一致性

### 复杂度控制

- 新增功能必须评估复杂度
- 如果违反简单性原则，必须在plan.md中说明理由
- 定期审查代码库，移除过时或冗余的功能

### 质量标准

- 所有代码提交必须通过基本功能测试
- 核心功能修改必须在实际游戏环境验证
- 性能回退必须有明确说明和解决计划
- Bug修复必须验证不引入新问题
