# Feature Specification: 单实例运行功能

**Feature Branch**: `002-single-instance`
**Created**: 2025-12-26
**Status**: Draft
**Input**: User description: "我需要你添加单实例运行功能。如果用户短时间内点击多次，会在托盘中出现2个程序图标"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 防止重复启动程序实例 (Priority: P1)

作为用户，当我启动程序时，如果程序已经在运行，系统应该阻止创建新实例并将焦点切换到现有实例，这样我就不会看到多个托盘图标，避免资源浪费和混乱。

**Why this priority**: 这是核心功能，直接解决用户报告的问题（短时间内多次点击导致多个托盘图标），防止资源浪费和用户体验混乱。

**Independent Test**: 可以通过以下方式独立测试：启动程序 → 再次双击程序图标 → 验证只有一个程序实例在运行，只有一个托盘图标显示，现有窗口获得焦点。

**Acceptance Scenarios**:

1. **Given** 程序未运行，**When** 用户首次启动程序，**Then** 程序正常启动，创建进程和托盘图标
2. **Given** 程序已在运行，**When** 用户再次双击程序图标，**Then** 不创建新进程，现有实例的主窗口获得焦点并置顶
3. **Given** 程序已在运行，**When** 用户短时间内（1秒内）多次点击程序图标，**Then** 只有第一次启动生效，后续点击被忽略，只显示一个托盘图标

---

### User Story 2 - 用户友好的重复启动提示 (Priority: P2)

作为用户，当我尝试启动已运行的程序时，系统应该给我清晰的视觉反馈（例如窗口闪烁或弹出提示），让我知道程序已经在运行，避免我误以为程序启动失败。

**Why this priority**: 提升用户体验，让用户明确知道程序状态，减少困惑。但优先级低于核心的单实例功能。

**Independent Test**: 可以通过以下方式测试：启动程序 → 再次点击程序图标 → 验证是否有明显的视觉反馈（窗口置顶、闪烁或通知）告知用户程序已在运行。

**Acceptance Scenarios**:

1. **Given** 程序已在运行且主窗口最小化，**When** 用户再次启动程序，**Then** 主窗口从最小化状态恢复并置顶显示
2. **Given** 程序已在运行且主窗口被其他窗口遮挡，**When** 用户再次启动程序，**Then** 主窗口置顶到所有窗口之上
3. **Given** 程序已在运行且仅显示托盘图标（无主窗口），**When** 用户再次启动程序，**Then** 托盘图标闪烁或弹出气泡通知提示"程序已在运行"

---

### User Story 3 - 异常情况下的单实例恢复 (Priority: P3)

作为用户，当程序异常退出后（如崩溃），系统应该能够正确清理单实例标记，让我可以正常重新启动程序，而不会被错误地阻止。

**Why this priority**: 确保单实例机制的健壮性，避免因异常退出导致程序无法启动。优先级较低，因为这是边缘情况。

**Independent Test**: 可以通过模拟测试：启动程序 → 强制终止进程（kill -9 或任务管理器结束进程）→ 再次启动程序 → 验证程序可以正常启动。

**Acceptance Scenarios**:

1. **Given** 程序正在运行，**When** 用户通过任务管理器强制结束进程，**Then** 单实例锁被自动释放
2. **Given** 程序异常崩溃退出，**When** 用户再次启动程序，**Then** 程序能够检测到旧锁已失效并正常启动
3. **Given** 程序正常退出，**When** 退出过程完成，**Then** 单实例锁被正确清理

---

### Edge Cases

- 当两个用户同时在不同用户账户下启动程序时会发生什么？（应该允许，单实例限制应该是每用户）
- 当程序在启动过程中（未完成初始化）被再次启动时会发生什么？（应该阻止，直到第一个实例完成启动）
- 当单实例锁文件损坏或权限不足无法访问时会发生什么？（应该记录警告日志，尝试降级启动或安全退出）
- 当用户从不同路径（如从快捷方式和原始exe）启动程序时会发生什么？（应该识别为同一程序，仍然单实例）
- 当操作系统重启但单实例锁未清理时会发生什么？（应该检测进程是否实际存在，清理失效锁）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须检测是否有程序实例已在运行
- **FR-002**: 系统必须在只有一个实例时才允许程序完全启动
- **FR-003**: 系统必须在检测到重复启动时将焦点切换到现有实例
- **FR-004**: 系统必须确保单实例检测在程序启动早期执行（在创建UI和托盘图标之前）
- **FR-005**: 系统必须在程序正常退出时清理单实例标记
- **FR-006**: 系统必须能够检测到失效的单实例标记（如进程已不存在）并自动清理
- **FR-007**: 系统必须为每个操作系统用户账户提供独立的单实例控制（用户A的实例不影响用户B）
- **FR-008**: 系统必须在重复启动时提供明显的视觉反馈（窗口置顶、闪烁或通知）
- **FR-009**: 系统必须记录单实例检测相关的关键事件到日志（启动、检测到重复、清理锁等）

### Assumptions

- 程序的单实例限制是针对每个操作系统用户账户，不是全局的（不同用户可以同时运行）
- 单实例检测应该基于应用程序标识符（如应用名称+用户ID），而不仅仅是进程名
- 用户期望的"现有实例获得焦点"是指主窗口或设置窗口置顶显示
- 程序可能在后台运行（只有托盘图标，无主窗口），此时的焦点切换应该是显示托盘通知
- 操作系统重启会清理大部分进程间通信机制（共享内存、命名管道等）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户短时间内（5秒内）多次点击程序图标，系统中只启动一个程序进程
- **SC-002**: 托盘区域始终只显示一个程序图标，无论用户点击多少次
- **SC-003**: 当用户尝试重复启动时，现有实例的窗口在1秒内获得焦点并置顶
- **SC-004**: 程序异常退出后，用户能够在5秒内成功重新启动程序（单实例锁自动恢复）
- **SC-005**: 单实例功能在 Windows 7/8/10/11 和 Linux 系统上都能正常工作
- **SC-006**: 由于单实例机制失败导致的用户支持工单减少 100%（从当前的多实例问题到零）
